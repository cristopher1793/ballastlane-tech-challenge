## Architecture: Clean Architecture (strictly enforced)

Structure the solution into the following projects:

1. `TaskApp.Domain` — Entities, interfaces, enums, domain exceptions. No dependencies.
2. `TaskApp.Application` — Business logic, validation, service interfaces, DTO definitions.
   Depends only on Domain.
3. `TaskApp.Infrastructure` — MongoDB repositories, JWT generation, password hashing,
   current user service. Implements Domain interfaces. Depends on Domain.
4. `TaskApp.API` — ASP.NET Core Web API, controllers, middleware registration.
   Depends on Application + Infrastructure.
5. `TaskApp.Tests` — xUnit unit tests for Application layer (Moq),
   integration tests for repositories (real MongoDB test database).
6. `taskapp-frontend` — React + TypeScript frontend (separate folder, Vite).

  - `status` (enum: Pending, InProgress, Completed)

- POST /api/auth/login           → Login, returns JWT (public)
- GET  /api/auth/me              → Get current user profile (authorized)
- GET  /api/auth/ping            → Health check (public, non-authorized)
- POST /api/auth/unlock/{userId} → Manually unlock a locked account (Admin role only)

- Due date cannot be in the past on creation
- Users can only access/modify their own tasks

## Architecture & Design Standards

### 1. Strongly Typed UserRole Enum
Do NOT use string for roles. Define in Domain layer:

  public enum UserRole { User, Admin }

The User entity must use:
  public UserRole Role { get; set; }

Never use magic strings like "Admin" or "User" anywhere in the codebase.

---

### 2. JWT Token Generation Abstraction
Application layer defines:

  public interface IJwtTokenGenerator
  {
      string GenerateToken(User user);
  }

Infrastructure layer implements:
  public class JwtTokenGenerator : IJwtTokenGenerator

UserService depends on IJwtTokenGenerator — it must NOT contain
any JWT creation logic directly. JWT logic belongs in Infrastructure.

---

### 3. Current User Abstraction
Application layer defines:

  public interface ICurrentUserService
  {
      string UserId { get; }
      string Username { get; }
      UserRole Role { get; }
  }

API layer implements:
  public class CurrentUserService : ICurrentUserService
  (reads from IHttpContextAccessor claims)

All business services must obtain the authenticated user through
ICurrentUserService — never read HttpContext claims directly in
controllers or services.

---

### 4. Repository Tests — Integration over Mocks
Do NOT mock IMongoCollection<T> for repository tests. Instead:
- Use a dedicated test MongoDB database (e.g. "taskapp_test")
- Write integration tests that run against a real MongoDB instance
- Tests must cover: Insert, Read, Update, Delete, user lookup,
  and unique constraint behavior (duplicate email/username)
- Spin up and tear down test data in test fixtures using IAsyncLifetime

## Security: Account Lockout

Implement account lockout logic in the Application layer (UserService),
tracked in the `users` MongoDB collection.

Rules:
- On each failed login attempt, increment `failedLoginAttempts` for that user
- After 3 consecutive failed attempts, set `isLocked = true` and record `lockedAt`
- Locked accounts are locked PERMANENTLY until manually unlocked — no automatic
  time-based unlock
- On a successful login, reset `failedLoginAttempts` to 0
- If a locked account attempts to log in, return HTTP 403 with message:
  "Account is locked due to too many failed attempts. Please contact an administrator."
- The unlock endpoint POST /api/auth/unlock/{userId} requires UserRole.Admin claim
- On unlock, reset `isLocked = false`, `failedLoginAttempts = 0`, `lockedAt = null`

Write unit tests for:
- Lockout triggers after exactly 3 failed attempts
- Login blocked when isLocked is true
- Successful login resets the counter
- Unlock endpoint returns 403 for non-Admin users
- Unlock resets all lockout fields correctly

## Security: Rate Limiting

Use .NET 8's built-in System.Threading.RateLimiting middleware
(Microsoft.AspNetCore.RateLimiting). No third-party packages.

Two named Fixed Window policies:

1. "auth-policy" — applied to all /api/auth/* endpoints:
   - 10 requests per minute per IP address
   - On limit exceeded: HTTP 429 with JSON:
     { "error": "Too many requests. Please wait before trying again." }

2. "general-policy" — applied globally to all other endpoints:
   - 60 requests per minute per IP address
   - Same JSON 429 response format

Implementation notes:
- Partition key: httpContext.Connection.RemoteIpAddress
- Register both policies in Program.cs via builder.Services.AddRateLimiter
- Apply "auth-policy" via [EnableRateLimiting("auth-policy")] on AuthController
- Apply "general-policy" as the default policy on app.UseRateLimiter()
- Configure OnRejected to return JSON with Content-Type: application/json
  (not the default plain text response)

Write unit/integration tests for:
- RateLimiter middleware is registered correctly
- Auth endpoints return 429 after exceeding the limit
- Correct JSON body is returned on 429 responses
